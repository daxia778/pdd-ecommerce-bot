# 🎨 UI Review Checklist (企业级 SaaS 风格规范自查表)

在修改或新增前端组件（Vue / CSS / HTML）后，请务必对着本清单自查。如果有任何一项未达标，你需要修改代码。

## 1. 🌈 颜色与材质 (Colors & Materials)
- [ ] **是否滥用了渐变？** (如 `bg-gradient-to-*`)
   - 渐变仅可用于：登录大背景修饰、个别的品牌主标识、极其重要的核心按钮（如"登录"、"确认"）。
   - **禁止**：侧边栏、常规数据监控卡片、普通按钮、表格标头、Tag/徽章 使用任何渐变色。
- [ ] **品牌色是否溢出？** (如 `text-red-600` 满天飞)
   - 重要信息/状态建议用标准的中性色或指定的语义色（如绿色代表成功，黄色代表警告）。
   - 品牌色 `#E02E24` 及邻近色仅用于：主按钮、Logo、链接、或者**极其重要**的指标提示。
- [ ] **背景层级清晰吗？** 
   - 页面背景应该是最深色的浅灰（`bg-gray-50` 或 `bg-gray-100`）。
   - 内容卡片必须是纯白（`bg-white`）。
   - 各级容器对比度（Contrast）是否够用，或边框（`border-gray-100/200`）是否正确。

## 2. 🧊 形状与尺寸 (Shapes & Sizes)
- [ ] **圆角是否超过了限制？** 
   - `rounded-2xl`(16px) 及以上，只应当用于大容器包裹外层。
   - 所有卡片面板一律限制在 `rounded-lg`(8px) 或最多 `rounded-xl`(12px)。
   - 按钮 `rounded` / `rounded-md`。
- [ ] **是否滥用了巨型内边距/外边距？** 
   - `p-8` 或更大。除某些宽屏的容器填充或者 Empty States 之外，一般用不上，应限制在 `p-4` 到 `p-6`。
- [ ] **元素大小是否影响可用性？**
   - 比如字号太小 `text-[8px]` 会导致很多用户阅读困难。限制最小可用字号为 `text-xs` (12px)。

## 3. 🌗 阴影与特效 (Shadows & Effects)
- [ ] **是否出现彩色发光？** 
   - `shadow-[0_0_8px_rgba(...)]`（各种带颜色的投影）- **强制移除**，替换为标准的 `shadow-sm`。
   - 任何 `shadow-red-500` - 替换成标准的黑白灰透明度的阴影。
- [ ] **动效滥用？** 
   - `animate-bounce`、`animate-pulse`（除个别在线绿点指示灯外）、`animate-float` 等花哨动效。若是管理后台面板（非登录展示页），**强制移除**。
   - `hover:scale-[1.02]` 在非展示型交互中通常很唐突。替换为 `hover:-translate-y-0.5` 与微小阴影或禁止修改卡片形变。

## 4. 📝 术论文案 (Terminology)
- [ ] **AI 浓度过高？**
   - 按钮文案有没有充斥了诸如 "AI 正在深度解析...", "AI 提纯...", 等字眼。
   - 请改为中性、面向结果的文案。如 "解析语义", "提取需求", "系统运行中", "数据加载中".
- [ ] **技术词汇直接抛给业务员？**
   - 如 `"JIT", "LLM", "SQLite争用", "TTFT"`，如果不是专门为"开发者模式"准备的探针面板，而是业务人员面板，**不允许出现**。请降级为 "系统状态", "响应耗时", "处理能力"。

## 5. 🏗️ 组件与布局 (Components & Layout)
- [ ] **是不是造了没必要的奇观？**
   - 比如，不画真实的手机边框效果，而是采用普通的预览窗口。
   - Sidebar 只有一种状态，不要画复杂的纹理。
- [ ] **有没有硬编码色值 / 宽高值？**
   - 尽量使用 tailwind 的配置体系 `w-64`, `min-h-[200px]` 优于 inline style，避免不可维护。

**结语**：如果你在代码里看到了违背本自查表的逻辑，请**果断删掉/重构**相关的 Tailwind 类。我们追求**专业、安静、稳重**的 B 端使用体验。
